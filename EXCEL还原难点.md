# Excel(.xlsx)网页还原难点

> 本文记录"从零做一个网页版 Excel 高保真预览器"过程中关于**为什么"完美还原微软"难**的核心认知,以及实际踩到的坑。
> 一句话主线:**OOXML 是"序列化格式",不是"渲染引擎";标准能保证你把文件读对,但不保证文件告诉了你渲染所需的全部信息。**

---

## 一、最大的误解:"标准协议" ≠ "实现轻松"

OOXML 的正式标准是 **ECMA-376 / ISO/IEC 29500**,六千多页。但要分清它**管什么、不管什么**:

| 标准**管**(可独立实现,有唯一答案) | 标准**不管**(implementation-defined / 留给应用) |
|---|---|
| XML 结构、元素/属性语义 | 字体怎么光栅化(DirectWrite/FreeType/CoreText 各不同) |
| 数字格式代码语法(§18.8.30) | 文字断行 / 连字 / 字距 |
| 样式继承、单元格类型、公式语法 | **自动行高**怎么算 |
| 单元格的值、格式 id、合并范围 | `General` 格式按列宽决定显示精度 |
| 主题色结构、关系(rels)寻址 | 抗锯齿、网格线颜色、默认字体、像素级宽度 |

→ 两个程序读同一文件,能对"B4 的值=719880、格式=`#,##0`、字体=Calibri 11"达成一致;
→ 但**画出来的像素**,标准从不保证一致。

---

## 二、渲染为什么"故意"不被标准化

像素级渲染要统一,标准就得连**整套字体引擎 + 图形栈**一起规定死,这在跨平台下不可能:

- 同一个 Calibri,Windows DirectWrite、Linux FreeType、Mac CoreText 渲染出的字形宽度、行距都不同
- 断行/字距依赖文字塑形引擎
- 自动行高、自适应宽度都依赖"测出来的字形宽度"

**类比**:CSS 是标准,但 Chrome 和 Firefox 不会逐像素相同。而 OOXML 对渲染的约束**比 CSS 还少得多**。

---

## 三、"微软 Office 的结果算标准吗" —— 不算,它是会动的靶子

Office 是**事实参考实现(de facto reference)**,不是形式标准:

- **本身不稳定**:同一个 .xlsx,Excel 2016 vs 365、Win vs Mac、不同 DPI、缺不缺某字体,画出来都可能不同。**两台装 Excel 的机器都未必像素一致。**
- **没被完整公开**:自动行高/自适应算法没有文档到能逐像素复刻。
- **会变**:微软自己在版本间改渲染。

**著名的自我打脸**:ECMA-376 标准化时(2007-2008),文里有些地方直接写"模拟 Word 95 的某某行为"(如 `autoSpaceLikeWord95` 兼容开关),把定义指向某个具体微软老应用——恰恰证明那部分**没有可独立实现的标准**,标准自己承认"这里照微软做"。

---

## 四、解析层的典型坑(标准虽定义清楚,但繁琐易错)

这些是"标准管了、但实现起来一堆边界"的部分。成熟库(ExcelJS/SheetJS)已踩平,自己手写则要逐个亲历:

1. **单元格 7 种类型靠索引**:`t="s"`=共享字符串池索引(非字符串本身)、`str`/`inlineStr`/`n`/`b`/`e`/`d`。富文本还拆成 `<r>` 多段独立样式。
2. **样式多级索引**:cell→`cellXfs`→`numFmtId/fontId/fillId/borderId/xfId`→`cellStyleXfs` 二级继承。
3. **数字格式是 mini 语言**:四段(正;负;零;文本)、`[Red]` 颜色、`[>=100]` 条件、千分位、`yyyy"年"`、`[h]:mm` 经过时间、分数、科学计数。
4. **日期是序列号**:含 Excel 故意保留的 **1900 闰年 bug**(序列号 60 = 不存在的 1900-02-29)+ 1904 系统两套基准。
5. **颜色二次计算**:`theme="4" tint="-0.25"` 要去 theme1.xml 取基色 + HSL 明暗运算;`indexed` 又是另一张调色板。
6. **一切靠 rels 间接寻址**:sheet 名→r:id→`workbook.xml.rels`→真实路径;图片/超链接/图表全靠二次跳转。
7. **单位换算反直觉**:列宽是"字符数"经 MDW 公式换算成像素;行高是 point。
8. **图表/图片 = DrawingML 另一套大 schema**:`xl/drawings/` 锚点 + `xl/charts/` 独立复杂 schema。
9. **命名空间 + 严格/转换式双版本**;`.xls` 更是完全不同的二进制 BIFF,与 OOXML 无关。

## 五、本项目实际踩到的 bug(都是"标准≠渲染"的具体落点)

| 缺陷 | 根因 | 类别 |
|---|---|---|
| 数字格式引擎 4 个 bug(双负号 `--42`、纯字面量段输出多余数字、文本段 `@` 不生效、`h:mm` 分钟误判为月份) | mini 语言自己实现,边界多 | 解析/格式化(可彻底修) |
| **列宽每列偏窄 5px** | 列宽公式漏了单元格内边距;`px = floor(width×MDW + 0.5) + 5`,默认 8.43 字符=64px(漏 +5 算成 59px) | 渲染换算(可彻底修) |
| 数据区右/下一片纯白、没网格 | 网格只画到数据边界;Excel 网格"视觉无限",数据外照样铺空格子 | 渲染行为(可彻底修,外推铺满视口) |
| **自动行高可能裁切文字** | 见下节,标准的"派生量"坑 | 部分可修(像素级不可消除) |

### 列宽 5px 的连锁反应
列宽偏窄 → 合并单元格(如 B4:J4)累计窄约 45px → 同一段文字被挤多折一行 → 行高(按 Excel 换行存的)装不下 → **裁切**。**一个换算误差会顺着布局链放大。**

---

## 六、"派生量"坑:自动行高(标准故意把活儿留给渲染器)

`<row>` 上跟高度相关只有两个属性:

```xml
<row r="4" ht="30" customHeight="1">   <!-- 用户钉死 30pt,权威,直接用 -->
<row r="5" ht="28.5">                   <!-- Excel 自动算的快照,无 customHeight -->
<row r="6">                             <!-- 啥都没存,要自己按内容算 -->
```

三层坑:
1. **有时根本没存 `ht`**(库生成的文件常省略)→ 标准把"算高度"甩给渲染程序,你得自己复刻 Excel 的换行+行距算法。
2. **存了也可能对不上**:`ht=28.5` 是 Excel 用**自己的字体度量**算"刚好放 2 行"的高度;你的 canvas `measureText` 字体度量略有差异 → 可能折成 3 行 → 装不下 → 裁切。**读对了高度照样穿帮。**
3. **这正是"完美还原微软"的不可达部分**:Excel 自动行高算法(ascent/descent、leading、padding 的精确组合)没公开到能逐像素复刻。

> 这跟列宽 bug 同根:文件里存的行高,是跟 **Excel 的换行结果**配套的;我的换行一旦不同,配套就崩。

---

## 七、结论:哪些可达 / 哪些不可达

- ✅ **可达(有标准、有唯一答案)**:把文件读对、值/格式/样式/合并/条件格式/结构解对。这是工程量问题,不是理论障碍。
- ✅ **可达(渲染行为,逼近)**:数字格式、列宽行高换算、网格、边框、填充、冻结、图片/图表近似——能做到常见环境下肉眼接近 Office。
- ❌ **不可达(理论上无标准答案)**:逐像素等于某台机器上的某版 Office。因为 Office 的渲染依赖字体栈/DPI/版本,**它自己都不保证两台机器一致**,自然没有"正确像素"可对。

> **遵守 ECMA-376** = 把文件读对,可达。
> **逐像素对齐 Office** = 追一个会动、依赖环境、没完全公开的参考实现,理论上不是一个有标准答案的目标。
> 我们能做的是**无限逼近**,并诚实标注哪些是近似。
